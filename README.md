# 📰 Daily Card News — 데일리 브리프

매일 아침, 다양한 분야의 뉴스를 큐레이션해 **카드뉴스 형태의 HTML**로 발행하는 프로젝트입니다.
AI 멀티 에이전트 파이프라인("하네스" 구조)으로 제작되며, GitHub에 매일 아카이빙됩니다.

**🔗 Repo**: [nonojin99/Daily-Card-News](https://github.com/nonojin99/Daily-Card-News)

## 발행 체계

매일 **2개**의 브리프가 발행됩니다.

### 1. 메인 브리프 (매일)
그날의 뉴스를 **화제성 우선**으로 선별한 종합 브리프.
분야 수는 고정하지 않고 그날의 뉴스 밀도에 따라 4~10개 카드로 유동 구성합니다.

### 2. 서브 브리프 (요일 로테이션)
한 분야를 깊게 다루는 전문 브리프. 화제성으로 거르지 않고 그 분야를 고르게 소개합니다.

## 제작 파이프라인 — "하네스" 구조

```
리서치 하네스 (Haiku)   →  분야별 정보 수집·요약
편집자 하네스 (Haiku)   →  검수·팩트체크·분량 조정
마케팅 하네스 (Sonnet)  →  "마케터의 시선" 인사이트 부착
디자인 하네스 (Sonnet)  →  카드뉴스 HTML 렌더링
검토 하네스 (Sonnet)   →  발행 전 최종 검증 (아래 체크리스트 통과)
```

> **검토 하네스의 한계 (정직하게)**: 이 단계는 별도의 독립된 검토자가 아니라,
> 같은 작업 세션이 발행 직전에 한 번 더 자기 결과를 점검하는 '강제 통과 관문'이다.
> 완전한 제3자 검토는 아니지만, 정합성·날짜·반복 같은 기계적 오류는 대부분 여기서 걸러진다.
> 특히 중요한 발행은 완성본을 새 대화에 붙여넣어 별도로 검토받는 것을 권장한다.

### 발행 전 검증 체크리스트 (검토 하네스)

브리프를 내보내기 전, 아래를 하나씩 통과시킨다. 하나라도 걸리면 고치기 전엔 발행하지 않는다.

1. **숫자 정합성** — 표·목록의 항목 수가 라벨과 맞나?
   (예: 토너먼트 라운드별 경기 수 — 16강=8경기, 8강=4경기, 4강=2경기. "8강인데 10팀" 같은 모순 금지)
2. **라운드·단계 표기** — 대회 진행 단계를 정확히 불렀나?
   (48개국 월드컵은 32강부터 토너먼트. 한 단계씩 밀려 부르지 않았는지 확인)
3. **날짜 최신성** — 모든 항목이 오늘 기준인가? 오래된 뉴스를 새것처럼 넣지 않았나?
4. **반복 필터** — 어제 카드와 실질적으로 겹치나? 변화가 있을 때만 재등장했나?
5. **한 사건 3카드** — 한 사건이 카드 3개를 넘지 않나?
6. **재난·민감 톤** — 참사·부고에 절제된 톤인가? 마케팅 인사이트가 부적절하지 않나?
7. **추측 금지** — 확인 안 된 수치·이름·라인업을 지어내지 않았나? 불확실하면 뺐나?
8. **변동 정보 경고** — 가격·환율·일정 등에 "⚠ 변동 가능" 표기를 했나?
9. **파일명 규칙** — `YYYY-MM-DD_{type}-brief.html` 형식이 정확한가?
10. **출처 링크 품질** — 모든 출처가 **기사 본문 URL**인가?
    (언론사 홈페이지·위키·블로그 링크 금지 · 같은 URL을 3개 이상 카드에 재사용 금지 · 위키/블로그는 언론사 원출처로 대체)
11. **요일 표기 검증** — 파일명의 날짜와 본문의 요일이 실제로 일치하는가? (`date` 명령으로 대조)

> 이 체크리스트는 실제 겪은 오류에서 나왔다.
> 1·2번은 '8강에 10팀' 오류(2026-07-07), 3번은 오래된 은하 뉴스,
> 4번은 전시 3일 반복, 7번은 미검증 선수 기록에서 비롯됐다.

## 연속성 트래커

매일 **메인 브리프 HTML 하단에 `<!-- tracker: ... -->` 주석**으로 두 가지를 기록한다:
① 추적 중인 스토리의 현재 상태(예: 월드컵=8강, 강진=사망 1,719명) ② 그날 다룬 항목 목록.

다음 날 브리핑은 GitHub에서 **어제 파일을 직접 읽고 시작**한다:

```bash
curl -s https://raw.githubusercontent.com/nonojin99/Daily-Card-News/main/YYYY-MM-DD_daily-brief.html
```

효과:
- 반복 필터·스토리 연속성이 대화 기억이 아닌 **파일 기반**으로 동작 (긴 대화에서의 기억 유실 방지)
- 어제 CSS를 그대로 재사용해 **디자인 드리프트 방지** (매일 기억으로 다시 쓰지 않음)

## 큐레이션 규칙

수 주간의 운영을 거치며 확립된 규칙입니다.

1. **화제성 우선** — 억지로 모든 분야를 채우지 않는다. 약한 분야는 과감히 제외.
2. **한 사건당 최대 3개 카드** — 빅이슈라도 갈래는 3개에서 끊는다.
3. **지속 정보는 변화가 있을 때만 재등장** — 전시·축제·시즌 정보는 종료 임박, 연장, 매진 등 새 변화가 있을 때만 다시 다룬다. 어제와 동일한 내용의 반복 금지.
4. **광범위 스캔 먼저** — 추적 중인 스토리에 매몰되지 말고, 매일 일반 뉴스를 넓게 훑어 새 대형 사건(재난 등)을 놓치지 않는다. 스캔은 "주요 뉴스+날짜" 검색보다 **주요 언론사 첫 화면을 직접 fetch**(연합뉴스·경향 등 1~2곳)하는 방식을 우선한다 — 검색은 오래된 기사가 섞여 나온다.
5. **오래된 뉴스를 새것처럼 다루지 않는다** — 날짜 확인이 안 되면 넣지 않는다.
6. **이미지·위성사진은 검증된 출처만** — 재난 이미지는 오정보가 흔하므로 Maxar Open Data, NASA Earth Observatory, Copernicus EMS 등 공식 출처만 신뢰.
7. **재난·참사에는 절제된 톤** — 마케팅 인사이트도 "침묵할 줄 아는 것"을 원칙으로.
8. **변동 정보에는 경고 표기** — 가격·일정·환율 등은 "⚠ 변동 가능, 링크 확인"을 명시.

### 서브 브리프 추가 규칙
- 각 항목에 **디테일 포인트**를 담는다: 작가·배경 소개, 관람 포인트, "알고 들으면 달라지는" 감상 포인트 등 (예: 볼레로의 스네어 드럼처럼, 알고 나면 다르게 보이는 지점).
- **신작(앨범·영화 등)의 디테일 포인트는 확인 불가한 세부를 지어내지 않는다.** 아직 들을·볼 수 없는 작품의 편곡·연출을 추측하는 대신, 검증 가능한 배경 지식(원곡·원작·모티프·작가의 스타일)에 근거해 감상의 열쇠를 준다.


1. **생성**: 매일 아침 Claude에게 "브리핑해줘" 요청 → 당일 뉴스 조사 → HTML 생성
2. **업로드**: 로컬 repo 폴더(`C:\Python Work\Daily-CARD-NEWS`)에 저장 후
   ```bash
   git add .
   git commit -m "YYYY-MM-DD daily brief"
   git push
   ```
3. **자동화(예정)**: Claude Cowork 스케줄 작업(매일 오전 7시)으로 생성→커밋→푸시 자동화

> GitHub 인증은 Git Credential Manager(브라우저 로그인)로 최초 1회 설정 완료.
> 이후 push는 무인증으로 동작합니다.
